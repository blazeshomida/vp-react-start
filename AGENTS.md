# Agent Instructions

## Project Overview

This repository is a React application starter built with Vite+, TanStack Start, TanStack Router, Nitro, Tailwind CSS, React Compiler, TypeScript, and Vitest.

The template is intentionally minimal and mostly unstyled. Keep starter code focused on durable app infrastructure, not example-app behavior. Placeholder route content, error UI, and not-found UI may stay plain unless the task explicitly asks for a UI pattern.

This repository follows Blaze's engineering handbook for reusable coding, documentation, workflow, and release standards.

Use this file for repository-specific instructions. Use the handbook as the canonical source for reusable standards unless this repository explicitly overrides them.

## Commands

Use Vite+ and the existing package scripts. Do not switch package managers.

```sh
# Install dependencies
vp install

# Run all readiness checks
vpr ready

# Run tests
vpr test

# Build
vpr build

# Run the app locally
vpr dev
```

Use `vp run ready` in GitHub Actions. Use `vpr ready` locally.

## Workflow

- Inspect the current repo state before changing files.
- Read the relevant files before implementing changes.
- Keep edits scoped to the requested task.
- Prefer existing project patterns over new abstractions.
- Do not rewrite existing files unless the requested behavior requires it.
- Do not revert user changes unless explicitly asked.
- Do not introduce unrelated formatting changes.
- Do not rename files, move modules, or change public APIs unless the task requires it.
- Check package scripts before inventing commands.
- Check existing tests before adding new test structure.
- Check generated file ownership before editing generated files.
- Summarize changed files, verification, and any remaining risk before handing work back.

## Project Structure

Important files and ownership boundaries:

- `vite.config.ts` is the Vite+ composition point.
- `tooling/format.ts` owns formatting config.
- `tooling/lint.ts` owns lint config.
- `tooling/patterns.ts` owns generated and output path patterns.
- `tooling/plugins.ts` owns the Vite plugin pipeline.
- `tooling/tasks.ts` owns the Vite+ task graph.
- `tooling/test.ts` owns Vitest config.
- `src/router.ts` owns TanStack Router creation and app router registration.
- `src/routes/__root.tsx` owns the root document, root route fallbacks, global CSS import, and router devtools.
- `src/site.config.ts` owns replaceable site metadata defaults.
- `src/lib/seo.ts` owns reusable metadata helpers.
- `src/routeTree.gen.ts` is generated by TanStack Router/Start.

## Type Safety

Follow the [TypeScript Standard](https://github.com/blazeshomida/handbook/blob/main/standards/code/typescript.md).

- Prefer TypeScript for application code.
- Avoid `any`.
- Do not use `as any`.
- Prefer `unknown` at untrusted boundaries.
- Fix types at the source instead of casting at call sites.
- Keep unsafe casts local, narrow, and justified by nearby runtime checks.
- Prefer `satisfies` for config objects and lookup maps.
- Use type-only imports and exports for type-only dependencies.
- Keep public API types named, readable, and stable.
- Prefer explicit return types for exported functions.
- Do not enable `verbatimModuleSyntax`; TanStack Start warns that it can cause server code to leak into client bundles.

## Code Style

- Follow existing project style first.
- Prefer small, focused modules.
- Prefer explicit public exports.
- Avoid accidental root exports.
- Prefer functional utilities unless a class is clearly the better model.
- Keep names descriptive and consistent with nearby code.
- Keep examples practical and minimal.
- Avoid clever abstractions that do not reduce real duplication.
- Omit comments that repeat the code.
- Keep comments focused on non-obvious behavior, constraints, or ownership.

## Template Boundaries

This is a starter template, not an example app.

Do not add these unless explicitly requested:

- auth placeholders
- database setup
- route loader examples
- server function examples
- design-system examples
- styled demo pages
- router default pending/error/not-found components
- deployment-specific Nitro presets
- a production `start` script tied to one hosting target

The starter should stay deployment-agnostic. Nitro is wired through `nitro/vite`, but no preset is forced.

## Routing

TanStack Router setup should stay close to the official Start shape:

- `src/router.ts` exports `getRouter()`.
- `src/router.ts` registers the router type through module augmentation.
- `src/routes/__root.tsx` owns the root document with `HeadContent`, `Outlet`, and `Scripts`.
- `src/routes/__root.tsx` imports global CSS as an intentional side effect.
- `src/routeTree.gen.ts` is committed for template convenience but treated as generated.

Router defaults currently include:

- `defaultPreload: "intent"`
- `defaultStructuralSharing: true`
- `scrollRestoration: true`

Do not add app-specific router options without a clear reason.

## Styling

Tailwind CSS is configured through `src/styles.css` and the Vite plugin.

`src/lib/utils.ts` provides the `cn()` helper using `clsx` and `tailwind-merge`.

`components.json` is configured for shadcn-style component generation using the existing aliases. Do not introduce a component library pattern unless the task explicitly asks for it.

## Metadata

`src/site.config.ts` contains starter defaults that should be replaced in real projects:

- site name
- title
- description
- production URL
- social image path

`src/lib/seo.ts` derives canonical URLs and social metadata from the site config and route path.

The default social image path is a placeholder. Add the actual public image asset or update the path before using the template in production.

## Testing And Checks

Follow the [Tooling Standard](https://github.com/blazeshomida/handbook/blob/main/standards/tooling.md).

- Run the smallest reliable checks that cover the change.
- Run broader checks when touching shared behavior, public APIs, build config, routing, metadata, or CI.
- If a check is not run, say so and explain why.
- If tests do not exist for the touched area, say that clearly and explain what was verified instead.

Examples:

- Type-only change: run `vpr ready` unless a narrower check is clearly enough.
- Routing change: run `vpr ready` and `vpr dev`.
- Build config change: run `vpr ready`.
- CI change: run `vpr ready`.
- Metadata helper change: run `vpr ready` and consider adding tests if behavior becomes non-trivial.

## Dependencies

Before adding a dependency:

- Check whether the repo already has an equivalent utility.
- Prefer small, well-maintained dependencies.
- Avoid adding dependencies for trivial logic.
- Explain why the dependency is needed.
- Update lockfiles consistently.

Use Vite+ dependency commands when adding packages. Do not switch package managers.

## Commits

Follow the [Commit Standard](https://github.com/blazeshomida/handbook/blob/main/standards/workflow/commits.md).

Before suggesting a commit message:

- Identify the primary change.
- Use the smallest useful scope.
- Use one primary intent.
- Avoid mixing unrelated changes.
- Mention breaking changes only when behavior or API compatibility actually breaks.

Do not create commits unless explicitly asked.

When giving commit commands, use this order:

```sh
# edit files first

vpr ready

git status --short
git add <files>
git commit -m "<message>"
```

Add extra verification before `git status --short` when the change needs it, such as `vpr dev` for runtime or routing changes.

## Pull Requests

Follow the [Pull Request Standard](https://github.com/blazeshomida/handbook/blob/main/standards/workflow/pull-requests.md).

A good PR summary should include:

- what changed
- why it changed
- how it was verified
- risks, tradeoffs, or follow-up work

For UI changes, include screenshots or describe the visual difference.

## Safety And Secrets

- Do not print secrets, tokens, private keys, or credentials.
- Do not commit `.env` files unless the repo intentionally tracks an example file.
- Prefer `.env.example` for documented environment variables.
- Redact sensitive values in logs and responses.

## Final Response

When work is complete, include:

- What changed.
- What was verified.
- Anything not run or not completed.
- Any follow-up that is required before merge or release.

Keep the response concise and specific to the completed work.
